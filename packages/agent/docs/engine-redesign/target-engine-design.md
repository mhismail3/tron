# Target engine design

Tron should become a live capability fabric for agents. The design borrows the
small primitive set from iii, but specializes it for local-first agent autonomy,
self-modification, traceability, authority, guardrails, and idempotent effects.

## Architecture overview

```text
agents / clients / cron / queues / streams / state / workers
        |
        v
live catalog + trigger registry + invocation router
        |
        v
policy engine + causal ledger + observability
        |
        v
in-process workers / external workers / sandbox workers
```

The engine owns the live catalog, invocation routing, trigger evaluation,
policy checks, catalog revisions, causal records, and observability. Workers own
behavior. Agents are not clients of a separate backend harness; they are actors
inside the same capability fabric as every other participant.

## First-principles constraints

The architecture is constrained by properties of the problem domain:

| Truth | Design constraint |
|-------|-------------------|
| Agents are stochastic actors. | Every action needs actor identity, authority, bounded execution, and causal records. |
| Capabilities change while agents work. | The catalog is live, revisioned, watchable, and queryable through stable meta-capabilities. |
| Side effects can be retried or replayed. | Mutating agent-visible functions require idempotency contracts and durable ledgers. |
| Schemas do not imply safety. | Function contracts include authority, effect class, risk, visibility, and policy metadata. |
| New code may be generated by agents. | Agent-created capabilities default to session visibility and require promotion for wider scope. |
| Trigger chains can loop. | Trigger execution carries depth, loop policy, dedupe keys, and budget limits. |
| Local-first still has many actors. | User, client, agent, worker, cron, queue, and system actions use one authority model. |
| Debugging needs causality. | Logs, events, queue jobs, state writes, and catalog mutations link into one causal graph. |

## Non-negotiable invariants

- The catalog is live. A frozen tool list may be cached for a provider call, but
  it is never the source of truth.
- Visibility is scoped. Session-created capabilities do not become workspace or
  system capabilities without an explicit promotion record.
- Invocation policy is non-bypassable for authority, visibility, ownership,
  and idempotency.
- Function ids are stable, and function revisions make contract changes
  explicit.
- Every mutating autonomous invocation has an idempotency key and effect class.
- Every trigger fire records source, target, authority, delivery mode,
  idempotency key, and causal parent.
- Worker cleanup removes only registrations owned by that worker or authority
  grant.
- Event sourcing remains the session ledger; state cannot silently replace
  reconstructable history.
- Every broader-visibility capability has provenance and a promotion path.
- Unknown or degraded worker health affects discovery and routing.

## Assumptions to minimize

The engine should not assume:

- workers tell the truth about schemas or effects;
- calls execute exactly once;
- network connections stay alive;
- the caller still has authority by the time work is dequeued;
- a function revision is compatible with a stale caller;
- the model saw every relevant catalog change in prompt context;
- retries are safe without function-specific idempotency;
- a generated worker should be trusted because a trusted agent created it.

Each of these is handled through metadata, validation, authority checks,
idempotency, catalog revisions, or explicit promotion.

## Known unknowns and containment

Some questions cannot be fully answered before implementation. They should be
contained by conservative defaults rather than hidden.

| Unknown | Containment |
|---------|-------------|
| How many live capabilities can an agent use effectively? | Expose search, ranking, filtering, and inspect flows instead of dumping the catalog into context. |
| How often will worker schemas lie or drift? | Track function revisions, validate payloads, record failures, and degrade or quarantine unhealthy capabilities. |
| Which generated workers deserve promotion? | Require provenance, tests, idempotency, risk metadata, and explicit promotion records. |
| Which catalog changes should interrupt an agent? | Use subscription classes chosen by task/session, not global interruption. |
| Can every external side effect be made idempotent? | Hide non-idempotent effects from autonomous agents or require approval/compensation wrappers. |
| How much authority should a spawned worker inherit? | Delegate the smallest namespace/function/trigger rights required, with expiry and revocation. |
| How complex can trigger cascades become? | Enforce causal-depth limits, loop detection, budgets, and dedupe keys. |
| Which state belongs in KV state versus the event ledger? | Default to event sourcing for session truth and use state only for coordination or derived/shared values. |

## Core primitives

### Worker

A worker is a live actor that can register functions and triggers.

Required properties:

- stable worker id;
- connection or instance id;
- kind: in-process, external, sandbox, agent, client, system, queue, stream,
  cron, state, MCP bridge, or compatibility;
- owner actor and delegated authority grant;
- namespace ownership;
- lifecycle state: starting, ready, degraded, draining, stopped;
- provenance: who created it, from what source, with which inputs, in which
  session/workspace;
- health and heartbeat metadata.

A worker is not necessarily a process. An in-process Rust module, a browser
tab, an agent-created sandbox, and a future remote process all fit the same
contract if they register capabilities and participate in causality.

### Function

A function is a capability contract registered by a worker.

Required properties:

- stable function id, preferably `namespace::operation`;
- function revision that increments on contract or handler changes;
- owner worker id;
- request schema;
- response schema or explicit opaque response marker;
- description and searchable tags;
- visibility scope: internal, session, workspace, system, client, worker,
  agent, admin;
- effect class;
- idempotency contract;
- required authority;
- risk level;
- allowed delivery modes;
- health state;
- cost and latency hints;
- provenance and promotion state.

Agent-visible mutating functions must declare idempotency. If a function cannot
provide an idempotency strategy, the engine must keep it out of autonomous
agent discovery or require an approval/confirmation wrapper that supplies the
missing safety.

### Trigger

A trigger is a causal rule that invokes a function.

Required properties:

- trigger id and revision;
- trigger type;
- target function id and required function revision policy;
- source condition/config;
- delivery mode: sync, void, enqueue;
- authority grant used for invocation;
- idempotency key strategy;
- loop/depth policy;
- retry/dead-letter policy when durable;
- visibility and owner worker;
- provenance and audit metadata.

HTTP routes, JSON-RPC compatibility calls, cron schedules, queue messages,
state changes, event-store appends, UI actions, approval responses, stream
events, and agent tool calls should all reduce to triggers.

The current runtime supports in-process `json_rpc` and `manual` trigger
dispatch plus queued delivery handoff. That is intentional: it proves trigger
metadata, authority, idempotency, revision, schema, queue receipts, and ledger
behavior before adding cron and remote external-worker delivery.

### Approval

Approval is a first-class primitive, not a UI-only confirmation path.

Required properties:

- approval id;
- target function id and payload fingerprint;
- original actor, authority grant, authority scopes, trace, parent invocation,
  trigger id, session/workspace, delivery mode, and idempotency key;
- status: pending, approved, denied, executed, failed;
- decision actor and timestamps;
- final result or structured error.

Agent-visible high-risk functions with approval-required authority metadata do
not run autonomously. The agent-facing invocation path creates a pending
approval, publishes an `approvals` stream event, and returns a structured
`APPROVAL_REQUIRED` envelope. Resolving the approval resumes the original
invocation with its original causal context and records the child result on the
approval record. Agents may request approvals but cannot resolve their own
approval gates; resolution is reserved for system/admin or user-authorized
actors with the approval scope.

### Engine

The engine is the live capability fabric. It owns:

- worker, function, trigger-type, and trigger registries;
- catalog revisions and catalog-change events;
- discovery/search/inspect APIs;
- invocation routing;
- policy enforcement;
- causal ledger records;
- trace propagation;
- idempotency ledgers;
- topology streams;
- approval records and approval lifecycle streams;
- health and observability surfaces.

The engine should not hide system liveness from agents. It should expose
liveness through stable meta-capabilities and make the consequences legible.

## Agent-native semantics

LLM provider APIs usually receive a fixed set of tools for one model call. Tron
should not solve live discovery by dumping the entire catalog into that static
tool list. Instead, the model gets a small stable set of meta-capabilities:

- discover or search capabilities;
- inspect a function/worker/trigger contract;
- invoke a capability;
- watch catalog changes;
- spawn a worker when authorized;
- promote a capability when authorized.

The underlying catalog is live. If a worker appears halfway through an agent
run, the agent can discover it on the next action step or receive a subscribed
catalog-change event and replan.

The first implementation surface is `EngineHost`. It registers the stable
`engine::*` meta-capabilities as normal catalog functions so agents can discover
them, while executing those functions through privileged host code that cannot
be overwritten by ordinary workers. `engine::invoke` is a delegated invocation:
the meta-function is stable, but the target function's current revision,
visibility, health, schema, authority, and idempotency policy are checked at
the child invocation.

Server startup owns this surface through an `EngineHostHandle`. The handle uses
the SQLite engine ledger at `engine-ledger.sqlite` beside the resolved event DB
path, and future adapters receive the handle through `RpcContext`. This makes
the engine lifecycle real without adding a client-facing API before the
compatibility mirror exists.

The agent capability client now treats approval-required high-risk functions as
pausable autonomous actions. A model can discover and inspect those functions,
but invocation produces a pending approval record instead of executing the
handler. This keeps the catalog live and agent-native without pretending that
all high-risk side effects are safe just because they have schemas and
idempotency keys.

## Local external workers

External workers begin loopback-only. A local worker can say hello, receive a
catalog snapshot, register session-visible volatile functions/triggers, send
heartbeats, and disconnect. Disconnect unregisters only that worker's volatile
catalog entries and emits normal availability changes. Workspace/system
visibility still requires explicit promotion, so agent-created local workers
do not silently become global capabilities.

Agents choose catalog-change subscriptions by task/session:

- worker spawned by this session became ready;
- function I invoked changed or disappeared;
- function under a namespace I care about appeared;
- high-risk capability promotion requested;
- dependency worker degraded;
- test worker finished registering functions.

Catalog changes should not automatically interrupt every run. The catalog is
live, and agents decide which change classes matter.

The initial watch model is cursor pull, not push streams. Catalog-change records
carry subject kind, change class, visibility, session id, and workspace id so a
watch request can resume from a revision and still avoid leaking hidden
session/internal entries. Push streams and backpressure belong in the later
stream worker.

Initial change-class meanings are intentionally agent-facing:

- `availability`: workers or functions appear, disappear, or otherwise become
  callable/non-callable;
- `contract`: a function contract changes without being newly added or removed;
- `trigger`: trigger types or trigger bindings change;
- `visibility`: a capability is promoted, demoted, or otherwise rescoped;
- `health`: routability changes without a contract change.

## Visibility and promotion

Live does not mean globally visible.

Default visibility rules:

| Creator | Default visibility | Reason |
|---------|--------------------|--------|
| Core daemon worker | System/internal, depending on function. | Stable built-in capabilities. |
| User/client action | Session or workspace, depending on action. | Preserve current local workflow expectations. |
| Agent-created worker/function | Session. | Enables self-modification with small blast radius. |
| Managed skill or repo-owned worker | Workspace or system after install policy. | Versioned source and user intent are available. |
| External untrusted worker | Hidden/quarantined until authorized. | Registration is not trust. |

Promotion is explicit and auditable:

1. Session-visible capability is created.
2. Agent or user tests it and records evidence.
3. Promotion function reserves the caller idempotency key, then evaluates
   schema, provenance, authority, tests, risk, and docs.
4. Engine changes visibility to workspace or system and emits a catalog event.

Promotion is a function call, not a side channel.
Duplicate promotion attempts with the same scoped idempotency key replay the
stored promotion result or fail as conflicts before a second target can mutate.

## Causality

Every action in the engine belongs to a causal graph.

Every invocation record should include:

- invocation id;
- trace id;
- parent invocation id, if any;
- actor id;
- authority grant id;
- session id and workspace id when applicable;
- worker id;
- function id and function revision;
- catalog revision observed at dispatch;
- trigger id and trigger revision, if trigger-caused;
- delivery mode;
- idempotency key;
- effect class and risk level;
- start/end timestamps;
- result status and structured error;
- downstream records created: events, logs, jobs, state writes, stream writes,
  spawned workers, trigger fires.

Catalog mutations should have equivalent causal records:

- worker registered/unregistered;
- function registered/updated/removed/promoted;
- trigger type registered/removed;
- trigger registered/updated/removed;
- visibility changed;
- authority grant created/revoked.

This is how live self-modification stays debuggable.

## Idempotency and effects

Every function must declare an effect class.

| Effect class | Meaning | Agent visibility |
|--------------|---------|------------------|
| Pure read | Reads state without mutation. | Allowed with authority. |
| Deterministic compute | Computes from payload only. | Allowed with schema. |
| Idempotent write | Mutates durable state but can safely repeat with same key. | Allowed with idempotency contract. |
| Append-only event | Adds immutable ledger/event entry. | Allowed with dedupe policy. |
| Reversible side effect | Effect can be undone by a compensating function. | Requires idempotency and compensation metadata. |
| External side effect | Calls external systems or devices. | Requires idempotency, approval or explicit authority, and audit. |
| Irreversible side effect | Cannot be safely undone. | Hidden from autonomous agents unless wrapped by explicit user approval. |

Every mutating agent-visible function must declare:

- idempotency key source: caller supplied, engine derived, trigger derived, or
  external provider key;
- dedupe scope: session, workspace, system, external service;
- replay behavior: return prior result, no-op, compensate, or reject;
- durable ledger location;
- retry compatibility.

The engine should reject autonomous agent invocation of mutating functions that
lack this contract.

## Guardrails as engine policy

Guardrails should not be scattered through handlers. They should be invocation
policy enforced by the engine and declared in function/trigger metadata.

Policy checks:

- schema validation before invocation;
- authority check before invocation;
- visibility filter before discovery;
- idempotency ledger check before mutation;
- approval trigger for high-risk effects;
- budget limits by session, actor, worker, and causal chain;
- rate limits by function/namespace/worker;
- causal-depth limits for trigger cascades;
- loop detection for state/event/trigger cycles;
- namespace ownership checks for registration;
- provenance checks for generated workers/functions;
- health checks before routing;
- output validation after invocation when response schema exists.

Policy itself should be expressible through functions and triggers where useful,
but the engine must have a small non-bypassable policy kernel for authority,
visibility, idempotency, and ownership.

## Live catalog

The catalog is a versioned live data structure.

Required behavior:

- every registration/removal/promotion increments a catalog revision;
- every change emits a catalog-change event;
- discovery can filter by actor, session, workspace, visibility, namespace,
  effect, risk, health, and text query;
- discovery includes enough metadata for agents to decide whether to inspect or
  invoke a function;
- invocation records the catalog revision used at dispatch;
- function ids are stable, but function revisions make contract drift explicit;
- stale references fail with structured errors that include replacement or
  re-discovery hints when available.

The live catalog is the mechanism that lets agents modify themselves while the
system is running.

## Built-in workers

Initial workers can all be in-process. The abstraction matters before process
isolation does.

| Worker | Responsibility |
|--------|----------------|
| `engine` | Discovery, catalog watch, health, authority metadata, promotion. |
| `rpc` | Transport compatibility worker for current JSON-RPC methods during migration. It owns the `json_rpc` trigger type and trigger bindings, not long-term business behavior. |
| `system` | Server status, ping, diagnostics, and future lifecycle/update capabilities. |
| `model` | Model catalog reads and future model/profile selection capabilities. |
| `settings` | Typed settings with iOS parity guarantees; currently fully generic-triggered for RPC compatibility. |
| `logs` | Local observability reads and append-only client-log ingestion; currently fully generic-triggered for RPC compatibility. |
| `prompt_library` | Prompt history/snippet reads and writes; currently fully generic-triggered for RPC compatibility. |
| `skills` | Skill registry and session-scoped active-skill state; currently fully generic-triggered for RPC compatibility. |
| `filesystem` | Home/list/read/create-dir capabilities now generic-triggered; broader file writes remain deferred until path authority and idempotent file mutation contracts are hardened. |
| `events` | Session/event-store append, read, reconstruct, subscribe/unsubscribe. All current `events.*` RPC methods are generic-triggered; subscribe/unsubscribe create stream subscription records while preserving current acknowledgements. |
| `session` | Session create/delete/fork/archive/unarchive/archiveOlderThan/export plus safe reads are generic-triggered canonical functions; `session.resume` remains handler-owned because it is still coupled to transport/session lifecycle. |
| `context` | Snapshot/audit/compaction/clear/canAcceptTurn methods are generic-triggered canonical functions with approval metadata on destructive commands. |
| `agent` | Queue controls are generic-triggered canonical functions; prompt execution, abort, subagent delivery, and confirmation/answer submission remain deferred until turn execution and approval streams are fully engine-owned. |
| `job` | Background/cancel/list/subscribe/unsubscribe are generic-triggered canonical functions; current background/cancel behavior is preserved while queue-backed job execution becomes the next hardening step. |
| `notifications` | Notification inbox read-state functions; currently fully generic-triggered for RPC compatibility. |
| `plan` | Session plan-mode state; currently fully generic-triggered for RPC compatibility. |
| `approval` | Pending/approved/denied/executed approval records for high-risk agent-visible invocations, with scoped approval lifecycle stream events. |
| `stream` | Durable subscriptions for catalog, session events, approvals, jobs, tool output, browser/display, transcription, notifications. |
| `state` | Scoped state for non-event-sourced data. |
| `queue` | Durable named queues, receipts, retries, cancellation, DLQ/redrive. |
| `cron` | Cron trigger type backed by current automations definitions and SQLite runtime state. |
| `tool` | Built-in tool capabilities and tool invocation. |
| `mcp` | MCP search/call and server lifecycle. |
| `worktree` | Git/worktree capabilities and conflict workflows. |
| `auth` | Provider auth, client auth, future worker tokens and grants. |
| `observability` | Logs, metrics, traces, causal graph queries, diagnostics. |

During the compatibility period, old JSON-RPC clients keep their stable method
names as `json_rpc` trigger ids and trigger config. Migrated methods execute as
canonical domain function ids such as `skills::activate` or `events::append`;
handler-only methods may still appear as internal `rpc::<method>` metadata so
the migration inventory remains complete. The end-state migration removes that
compatibility inventory after clients and agents invoke canonical domain
capabilities directly.

## State, streams, queues, and events

State is scoped shared data. It is not the session ledger. Use it for values
that do not need append-only reconstruction.

Streams are durable or resumable views over change records. WebSocket is a
transport, not the stream abstraction.

Queues provide at-least-once delivery. At-least-once means mutating functions
must be idempotent. Queue retries and DLQ movement must be visible in the
causal ledger.

Events remain the durable source of truth for sessions. The event store should
become an engine worker, not disappear into state.

## Security and authority

Auth is not just "is this connection allowed?" It is actor authority.

Authority records should include:

- actor identity;
- actor kind: user, agent, client, worker, cron, system;
- grant id;
- grant issuer;
- namespace registration rights;
- invocation rights;
- visibility rights;
- trigger registration rights;
- delegation rights;
- approval requirements;
- expiry/revocation.

Agent-created workers inherit narrowed authority. A parent agent can delegate a
specific namespace and function set, not its full authority.

## Client API transition

The final client API can break, but transition should be controlled:

1. Keep current `/ws` JSON-RPC while compatibility triggers preserve it.
2. Register every current method as a classified bridge spec; handler-only
   methods may keep internal `rpc::<method>` metadata, while migrated methods
   register canonical domain functions.
3. Move selected method behavior into domain-owned engine functions. Use
   method-specific thin adapters only as temporary parity scaffolding when a
   method group is not ready for generic dispatch.
4. Replace thin adapters in a method group with `json_rpc` triggers targeting
   canonical domain functions as soon as tests prove parity.
5. Add live catalog discovery and catalog streams behind the server.
6. Move selected client flows to engine-native calls once stable.
7. Remove compatibility handlers only after Mac/iOS clients consume the new
   contract.

Compatibility is for migration and validation, not a permanent second system.

## Design review checklist

Every future engine change should answer:

- Which actor and authority grant can cause this?
- What visibility scope does it enter by default?
- What effect class does it have?
- What idempotency contract prevents duplicate side effects?
- What causal records are written before and after execution?
- What catalog revision or function revision does the caller rely on?
- What happens if the worker disconnects, restarts, or lies about its schema?
- What happens if the trigger fires twice or creates a loop?
- How can the capability be promoted, revoked, or quarantined?
